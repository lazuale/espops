# 05. Обновление

[Назад: тестовый стенд](04-dev-stand.md) · [К содержанию](../README.md) · [Далее: чек-листы](06-checklists.md)

Обновление делается по простому правилу: сначала резервная копия, затем проверка на тестовом стенде, только после этого рабочий стенд.

Не обновлять одновременно EspoCRM и MariaDB без необходимости. Если после обновления появится ошибка, так проще понять причину.

## 1. Какие бывают обновления

| Вид обновления | Что меняется | Как проверять |
|---|---|---|
| Обновление файлов проекта из GitHub | документация, compose-файлы, вспомогательные файлы | `docker compose config`, затем запуск |
| Обновление EspoCRM | тег `ESPOCRM_IMAGE` в `.env` | сначала тестовый стенд |
| Обновление MariaDB | тег `MARIADB_IMAGE` в `.env` | только после backup и проверки на тесте |

## 2. Обновление файлов проекта из GitHub

Эта процедура нужна, когда в репозитории изменились документация, compose-файлы или вспомогательные файлы.

Перед обновлением обязательно сделать резервную копию рабочего стенда:

```bash
cd /opt/espocrm
mkdir -p backups
bash ./scripts/backup-docker-container.sh espocrm ./backups
```

Проверить, что на сервере нет ручных изменений tracked-файлов:

```bash
git status --short
```

Если команда показывает изменения в `docker-compose.*.yml`, `README.md`, `docs/` или `scripts/`, сначала нужно разобраться с ними. На рабочем сервере такие файлы не правят вручную без последующего коммита в репозиторий.

Забрать изменения из GitHub:

```bash
git pull --ff-only
```

Проверить compose-конфигурацию:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml config
```

Если проверка завершилась с ошибкой, не применять обновление. Сначала исправить проблему.

Применить изменения:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d
```

Проверить состояние:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
curl -I http://10.0.2.7
```

## 3. Обновление версии EspoCRM или MariaDB

Версии задаются в `.env.prod` и `.env.dev`.

| Переменная | Что задаёт |
|---|---|
| `ESPOCRM_IMAGE` | образ EspoCRM |
| `MARIADB_IMAGE` | образ MariaDB |

Пример:

```env
ESPOCRM_IMAGE=espocrm/espocrm:9.3.7-apache-trixie
MARIADB_IMAGE=mariadb:11.4.11-noble
```

Порядок обновления версий:

1. Сделать резервную копию рабочего стенда.
2. Перенести рабочую копию в тестовый стенд по [`04-dev-stand.md`](04-dev-stand.md). После переноса тестовый стенд работает на текущей версии с данными рабочего стенда.
3. Изменить версию образа в `.env.dev` (`ESPOCRM_IMAGE` или `MARIADB_IMAGE`).
4. Запустить тестовый стенд командами ниже. При первом старте на новой версии EspoCRM сам обновит схему базы — это нормально и выполняется на тестовых данных, а не на рабочих.
5. Проверить работу тестового стенда по разделу 5.
6. Только после успешной проверки изменить версию в `.env.prod` и применить обновление на рабочем стенде.

Для тестового стенда:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml pull
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
docker compose --env-file .env.dev -f docker-compose.dev.yml ps
```

Если тестовый стенд не поднялся или работает неправильно, рабочий стенд не обновлять.

Для рабочего стенда после успешной проверки на dev:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml pull
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
```

## 4. Откат после неудачного обновления

Откат делается не “понижением версии наугад”, а восстановлением из резервной копии.

Если обновление рабочего стенда прошло неудачно:

1. Не выполнять дополнительные эксперименты на рабочем стенде.
2. Остановить пользователей.
3. Восстановить резервную копию по [`03-backup-restore.md`](03-backup-restore.md).
4. Разобрать ошибку на тестовом стенде.

## 5. Что проверить после обновления

| Проверка | Ожидаемый результат |
|---|---|
| `docker compose ps` | контейнеры запущены |
| Веб-интерфейс | сайт открывается |
| Вход администратора | вход работает |
| Основные разделы | данные доступны |
| Вложения | файлы открываются |
| Фоновые задания | нет явных ошибок в `espocrm-daemon` |
| WebSocket | порт доступен, нет явных ошибок в журнале |

Если после обновления что-то работает не так, причины и порядок диагностики собраны в [`07-troubleshooting.md`](07-troubleshooting.md).
