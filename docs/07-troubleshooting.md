# 07. Типичные проблемы

[Назад: чек-листы](06-checklists.md) · [К содержанию](../README.md)

Этот документ помогает быстро найти причину, когда стенд ведёт себя не так, как ожидается. Он не заменяет основные инструкции, а собирает в одном месте частые симптомы и понятный порядок действий.

Все команды для рабочего стенда выполняются из каталога проекта:

```bash
cd /opt/espocrm
```

Для тестового стенда команды те же, но с `--env-file .env.dev -f docker-compose.dev.yml` и именами контейнеров с суффиксом `-dev`.

## 1. Общий порядок диагностики

Прежде чем что-то менять, нужно собрать факты. В большинстве случаев достаточно трёх шагов.

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm
docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-db
```

Сначала смотрим состояние контейнеров, затем журнал `espocrm`, затем журнал базы. Не нужно повторно запускать установку и не нужно удалять тома: почти все проблемы решаются без потери данных.

## 2. Быстрый справочник: симптом и причина

| Симптом | Вероятная причина | Куда смотреть |
|---|---|---|
| Сразу после первого `up -d` сайт не открывается | стенд ещё инициализируется | раздел 3 |
| Контейнер `espocrm` долго в статусе `health: starting` | идёт установка или нет связи с базой | разделы 3 и 6 |
| Сайт открывается, но без стилей или уводит на другой адрес | несовпадение `siteUrl` | раздел 4 |
| Нет живых уведомлений, ошибки WebSocket в консоли браузера | порт или адрес WebSocket | раздел 5 |
| В журнале `espocrm` ошибки подключения к базе | пароль БД не совпадает или база не готова | раздел 6 |
| `up -d` падает с ошибкой про занятый порт | порт `80` или `8081` занят | раздел 7 |
| Скрипт backup пишет «Container is not running» | один из контейнеров остановлен | раздел 8 |
| Backup или восстановление застревает на файлах | нет образа `alpine` | раздел 9 |
| Тестовый стенд уводит на рабочий адрес | dev не переприменил свой `siteUrl` | раздел 10 |
| На рабочем стенде случайно выполнили `down -v` | удалены тома с данными | раздел 11 |

## 3. Сайт не открывается сразу после запуска

Первый запуск идёт дольше обычного: инициализируется база, затем `espocrm` выполняет установку и создаёт `data/config.php`, и только потом стартуют фоновый обработчик и WebSocket. Пока `espocrm` не дошёл до статуса `healthy`, сайт может не отвечать.

Что сделать:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
docker compose --env-file .env.prod -f docker-compose.prod.yml logs -f espocrm
```

Дождитесь в журнале сообщений об успешном завершении установки и статуса `healthy` у `espocrm`. Если статус не меняется несколько минут, переходите к разделу 6 — скорее всего, проблема в подключении к базе.

## 4. Сайт открывается, но без стилей или уводит на другой адрес

Так проявляется несовпадение адреса стенда (`siteUrl`) с тем, по которому вы реально открываете CRM. Браузер подгружает ресурсы и обращается к WebSocket по адресу из настроек, и если он не совпадает с фактическим — стили и скрипты не грузятся, появляются редиректы.

На этом стенде адрес задаётся через переменные `ESPOCRM_SITE_URL` и `ESPOCRM_WEB_SOCKET_URL` в `.env`, а в compose они применяются при каждом запуске через `ESPOCRM_CONFIG_SITE_URL` и `ESPOCRM_CONFIG_WEB_SOCKET_URL`.

Что проверить:

1. В `.env.prod` значение `ESPOCRM_SITE_URL` совпадает с тем адресом, по которому вы открываете CRM.
2. `ESPOCRM_WEB_SOCKET_URL` указывает на тот же хост и правильный порт.
3. После правки `.env` пересоздать контейнеры:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d --force-recreate
```

## 5. Не работает WebSocket

WebSocket нужен для живых уведомлений и обновления интерфейса. Если в консоли браузера видны ошибки подключения к `ws://...`, проверяйте порт и адрес.

| Что проверить | Команда или значение |
|---|---|
| Контейнер WebSocket запущен | `docker compose --env-file .env.prod -f docker-compose.prod.yml ps espocrm-websocket` |
| Порт WebSocket доступен с рабочего места | `Test-NetConnection 10.0.2.7 -Port 8081` |
| Адрес в настройках | `ESPOCRM_WEB_SOCKET_URL=ws://10.0.2.7:8081` в `.env.prod` |
| Журнал WebSocket | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-websocket` |

## 6. Ошибка подключения к базе

В журнале `espocrm` встречаются сообщения вроде «Unable to connect to the database» или код ошибки `2002`. Частые причины:

- База ещё не готова. Посмотрите статус `espocrm-db` — он должен быть `healthy`. Дайте базе время на инициализацию.
- Пароль БД в настройках не совпадает с паролем, который уже записан в томе. Пароли БД, имя базы и пользователь задаются при первой установке и сохраняются в `data/config-internal.php` внутри тома. Если позже поменять только `.env`, существующее подключение это не изменит.

Что сделать:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-db
```

Если стенд восстанавливали из backup, сверьте параметры БД по разделу о восстановлении в [`03-backup-restore.md`](03-backup-restore.md): имя, пользователь и пароль базы должны соответствовать тем, что записаны в восстановленных файлах.

## 7. Порт занят

Если `up -d` сообщает, что порт уже используется, значит на сервере что-то слушает порт `80` или `8081`.

```bash
sudo ss -lntp | grep -E ':80|:8081' || true
```

Дальше есть два пути: освободить порт, остановив занявшую его службу, либо изменить порт в `.env.prod` (`ESPOCRM_HTTP_PORT` или `ESPOCRM_WEBSOCKET_PORT`) и пересоздать контейнеры. При смене порта не забудьте поправить `ESPOCRM_SITE_URL` и `ESPOCRM_WEB_SOCKET_URL`, чтобы адреса остались согласованными.

## 8. Скрипт backup завершается с ошибкой

Скрипту нужны оба работающих контейнера: `espocrm` (из него читаются параметры базы) и `espocrm-db` (из него снимается дамп). Если выводится ошибка «Container is not running», проверьте состояние:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
```

Поднимите остановленный контейнер и повторите резервное копирование.

## 9. Backup или восстановление застревает на файлах

Работа с файловым томом EspoCRM идёт через образ `alpine`. Если сервер без доступа в интернет и образа нет локально, операция остановится на этапе файлов.

```bash
docker image ls alpine
```

Если образа нет, загрузите его заранее на сервере с доступом в сеть:

```bash
docker pull alpine
```

## 10. Тестовый стенд уводит на рабочий адрес

После переноса рабочей копии тестовый стенд должен открываться по своему адресу. Если он уводит на `http://10.0.2.7`, значит контейнер ещё не переприменил тестовый `siteUrl`.

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d --force-recreate
```

Убедитесь, что в `.env.dev` указан тестовый `ESPOCRM_SITE_URL` (например, `http://10.0.2.7:8080`). Подробнее о механике переноса — в [`04-dev-stand.md`](04-dev-stand.md).

## 11. На рабочем стенде случайно удалили тома

Команды `docker compose down -v`, `docker volume prune` и `docker system prune --volumes` удаляют тома, а вместе с ними базу и файлы EspoCRM. Если это произошло на рабочем стенде, данные нужно восстанавливать из резервной копии.

Порядок действий:

1. Не выполнять дополнительные команды, которые могут перезаписать или удалить что-то ещё.
2. Поднять стенд заново: `docker compose --env-file .env.prod -f docker-compose.prod.yml up -d`.
3. Восстановить данные из последней резервной копии по [`03-backup-restore.md`](03-backup-restore.md).

Чтобы такой ситуации не было, опасные команды перечислены в [`02-operations.md`](02-operations.md) в разделе о том, что нельзя делать на рабочем стенде.
