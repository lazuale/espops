# 02. Обслуживание рабочего стенда

[Назад: установка](01-install.md) · [К содержанию](../README.md) · [Далее: резервное копирование](03-backup-restore.md)

Этот документ нужен для повседневной работы с уже запущенным рабочим стендом: посмотреть состояние, открыть журналы, остановить или перезапустить контейнеры без удаления данных.

Все команды выполнять из каталога проекта:

```bash
cd /opt/espocrm
```

## 1. Из чего состоит рабочий стенд

Рабочий стенд состоит из четырёх контейнеров. Если что-то не работает, сначала нужно понять, к какому контейнеру относится проблема.

| Контейнер | Назначение | Когда смотреть |
|---|---|---|
| `espocrm-db` | база данных MariaDB | ошибки подключения к БД, резервное копирование, восстановление |
| `espocrm` | основной веб-контейнер EspoCRM | сайт не открывается, ошибки интерфейса |
| `espocrm-daemon` | фоновые задания EspoCRM | не выполняются фоновые задачи |
| `espocrm-websocket` | WebSocket | нет live-уведомлений, ошибки WebSocket |

Проверить состояние:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
```

В нормальном состоянии у `espocrm` и `espocrm-db` в колонке `STATUS` стоит `healthy`. Если один из контейнеров постоянно перезапускается или висит в `health: starting`, нужно смотреть его журнал.

## 2. Где хранятся данные

Данные не лежат в каталоге `/opt/espocrm`. Каталог проекта содержит только compose-файлы, шаблоны `.env`, документацию и скрипт резервного копирования.

Реальные данные находятся в Docker-томах:

| Том Docker | Путь внутри контейнера | Что хранится |
|---|---|---|
| `espocrm-prod_espocrm-db` | `/var/lib/mysql` | база данных MariaDB |
| `espocrm-prod_espocrm` | `/var/www/html` | файлы EspoCRM, загрузки, настройки и доработки |

Внутри `/var/www/html` особенно важны:

| Путь | Назначение |
|---|---|
| `data/config.php` | основной конфиг EspoCRM |
| `data/config-internal.php` | параметры подключения к базе (имя, пользователь, пароль) |
| `data/upload/` | загруженные файлы |
| `custom/` | серверные доработки |
| `client/custom/` | клиентские доработки |

Для полного восстановления нужны оба компонента: база данных и файлы `/var/www/html`.

## 3. Посмотреть журналы

Журналы нужны, когда контейнер не запускается, сайт не открывается или есть ошибки в работе CRM.

| Что посмотреть | Команда |
|---|---|
| Основной контейнер | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm` |
| База данных | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-db` |
| Фоновые задания | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-daemon` |
| WebSocket | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs --tail=200 espocrm-websocket` |
| Основной контейнер в реальном времени | `docker compose --env-file .env.prod -f docker-compose.prod.yml logs -f espocrm` |

Если непонятно, где ошибка, начинать с `espocrm`, потом смотреть `espocrm-db`. Частые причины и их признаки собраны в [`07-troubleshooting.md`](07-troubleshooting.md).

## 4. Остановить и запустить без удаления данных

Обычная остановка контейнеров не удаляет Docker-тома и не удаляет данные.

Остановить рабочий стенд:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml down
```

Запустить после остановки:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d
```

Проверить состояние:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
```

## 5. Автозапуск после перезагрузки сервера

Все контейнеры стенда запускаются с параметром `restart: always`. Это значит, что после перезагрузки сервера или сбоя Docker сам поднимет рабочий стенд — запускать его вручную не нужно.

После перезагрузки достаточно проверить, что всё поднялось:

```bash
cd /opt/espocrm
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
```

У `espocrm` и `espocrm-db` в колонке `STATUS` должно быть `healthy`. Сразу после загрузки сервера это может занять до двух минут. Если через несколько минут стенд так и не поднялся, посмотрите журнал `espocrm` и загляните в [`07-troubleshooting.md`](07-troubleshooting.md).

Отдельно проверьте, что включён автозапуск самого Docker (делается один раз при подготовке сервера):

```bash
systemctl is-enabled docker
```

Ожидается ответ `enabled`. Если ответ другой, включите автозапуск: `sudo systemctl enable --now docker`.

## 6. Пересоздать контейнеры без удаления данных

Эта команда нужна, если поменялись настройки compose или `.env.prod`, но удалять данные нельзя.

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d --force-recreate
```

После пересоздания проверить:

```bash
docker compose --env-file .env.prod -f docker-compose.prod.yml ps
curl -I http://10.0.2.7
```

## 7. Частые проверки

| Что проверить | Команда |
|---|---|
| Свободное место | `df -h` |
| Размеры Docker | `docker system df -v` |
| Все запущенные контейнеры | `docker ps` |
| Сайт с сервера | `curl -I http://10.0.2.7` |
| WebSocket-порт с Windows | `Test-NetConnection 10.0.2.7 -Port 8081` |
| Репозиторий Git | `git remote -v` |
| Текущая ветка | `git branch --show-current` |

## 8. Рост журналов и место на диске

Со временем Docker накапливает служебные журналы контейнеров. Это не база данных и не файлы EspoCRM, но при долгой работе они могут занять много места на сервере. Раздел построен от безопасных действий к тем, что затрагивают весь сервер.

### Как понять, что место заканчивается

Свободное место на диске:

```bash
df -h
```

Сколько места занимает Docker (образы, контейнеры, тома, журналы):

```bash
docker system df -v
```

Размер каталога с резервными копиями (если он уже есть):

```bash
du -sh /opt/espocrm/backups 2>/dev/null || echo "каталог backups ещё не создан"
```

### Что можно сделать безопасно

В первую очередь проверьте резервные копии — обычно место занимают именно они. Старые локальные архивы можно удалять, если они уже перенесены во внешнее защищённое место. Безопасная очистка старых копий описана в [`03-backup-restore.md`](03-backup-restore.md) (раздел про автоматическую копию и очистку): там же команда `find ... -mtime +30 -delete`, которая удаляет только архивы старше 30 дней.

Эти действия не затрагивают базу и файлы EspoCRM.

### Действие уровня сервера: ограничить размер журналов

Если место продолжает быстро заканчиваться и видно, что его занимают именно журналы контейнеров, можно один раз задать ограничение их размера. Это настройка уровня Docker-сервера, и применять её нужно осознанно: изменение требует перезапуска Docker, а перезапуск затрагивает **все** контейнеры на сервере, а не только EspoCRM.

На сервере, где кроме EspoCRM работают другие контейнеры, эту настройку лучше согласовать с тем, кто отвечает за инфраструктуру, или поручить ему.

Если сервер выделен под EspoCRM, ограничение задаётся в файле `/etc/docker/daemon.json`:

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

После изменения файла перезапустите Docker:

```bash
sudo systemctl restart docker
```

Важно понимать одну деталь: новое ограничение применяется только к контейнерам, **созданным после** изменения. У уже работающих контейнеров журналы продолжат расти по-старому до тех пор, пока контейнеры не будут пересозданы. Поэтому после перезапуска Docker пересоздайте стенд:

```bash
cd /opt/espocrm
docker compose --env-file .env.prod -f docker-compose.prod.yml up -d --force-recreate
```

Пересоздание безопасно для данных: тома с базой и файлами EspoCRM не затрагиваются. Сбрасываются только накопленные служебные журналы старых контейнеров — новые контейнеры начинают вести журнал с нуля и уже с ограничением размера.

### Чего нельзя делать ради освобождения места

Эти команды удаляют Docker-тома, а вместе с ними базу и файлы EspoCRM:

```bash
docker volume prune
docker system prune --volumes
docker compose down -v
```

Освобождать место такими командами нельзя. Полный список опасных команд — в разделе 9.

## 9. Что нельзя делать на рабочем стенде

Эти команды опасны, потому что могут удалить Docker-тома с базой и файлами EspoCRM.

```bash
docker compose down -v
docker volume prune
docker system prune --volumes
docker volume rm espocrm-prod_espocrm-db
docker volume rm espocrm-prod_espocrm
```

Также нельзя коммитить в Git реальные настройки и резервные копии:

```text
.env.prod
.env.dev
backups/
restore-tmp/
*.tar.gz
*.sql
*.log
```

Если нужно освободить место на сервере, сначала проверить, что именно занимает место, и только потом удалять лишнее. На рабочем стенде не использовать массовую очистку томов.
