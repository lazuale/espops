# 04. Тестовый стенд

[Назад: резервное копирование](03-backup-restore.md) · [К содержанию](../README.md) · [Далее: обновление](05-update.md)

Тестовый стенд нужен для проверки настроек, обновлений и доработок без риска для рабочего стенда. Его можно спокойно перезаписывать копией рабочего стенда и использовать для экспериментов.

Важно: операции с тестовым стендом не должны останавливать и менять рабочий стенд.

## 1. Запуск тестового стенда

Тестовый стенд запускается отдельным compose-файлом и использует отдельные контейнеры и Docker-тома.

| Параметр | Значение по умолчанию |
|---|---|
| Файл настроек | `.env.dev` |
| Compose-файл | `docker-compose.dev.yml` |
| Веб-адрес | `http://10.0.2.7:8080` |
| WebSocket | `ws://10.0.2.7:8083` |

Тестовый стенд рассчитан на работу на том же сервере одновременно с рабочим. Чтобы они не конфликтовали, в `.env.dev` используются другие порты, чем в `.env.prod`: веб-порт `8080` вместо `80` и WebSocket-порт `8083` вместо `8081`. Не копируйте порты из `.env.prod` в `.env.dev` — иначе оба стенда попытаются занять один и тот же порт.

Перед первым запуском dev убедитесь, что нужные порты свободны:

```bash
sudo ss -lntp | grep -E ':8080|:8083' || true
```

Создать настройки:

```bash
cd /opt/espocrm
cp .env.dev.example .env.dev
nano .env.dev
chmod 600 .env.dev
```

Запустить тестовый стенд:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
```

Проверить контейнеры:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml ps
```

Открыть с рабочего компьютера:

```text
http://10.0.2.7:8080
```

`localhost` использовать только если открываете dev прямо с самого сервера:

```text
http://localhost:8080
```

Для рабочих мест по сети используется адрес `http://10.0.2.7:8080`.

## 2. Контейнеры и тома тестового стенда

| Рабочий стенд | Тестовый стенд | Назначение |
|---|---|---|
| `espocrm` | `espocrm-dev` | веб-контейнер EspoCRM |
| `espocrm-db` | `espocrm-db-dev` | база MariaDB |
| `espocrm-daemon` | `espocrm-daemon-dev` | фоновые задания |
| `espocrm-websocket` | `espocrm-websocket-dev` | WebSocket |

| Тестовый том | Что хранит |
|---|---|
| `espocrm-dev_espocrm-db-dev` | база тестового стенда |
| `espocrm-dev_espocrm-dev` | файлы тестового стенда |

Удалять тома тестового стенда можно только осознанно. Это не затрагивает рабочий стенд, но полностью очищает тестовую CRM.

## 3. Подготовка dev для переноса рабочей копии

Если тестовый стенд будет регулярно обновляться копией рабочего стенда, настройки БД в `.env.dev` должны быть совместимы с рабочей копией.

Самый простой и стабильный вариант — использовать в `.env.dev` те же значения имени БД, пользователя и пароля, что и в `.env.prod`:

```env
ESPOCRM_DATABASE_NAME='espocrm'
ESPOCRM_DATABASE_USER='espocrm'
ESPOCRM_DATABASE_PASSWORD='CHANGE_ME_SAME_AS_PROD_DB_PASSWORD'
```

Адреса и порты при этом остаются тестовыми:

```env
ESPOCRM_SITE_URL=http://10.0.2.7:8080
ESPOCRM_WEB_SOCKET_URL=ws://10.0.2.7:8083
```

Здесь важно понимать, что́ при переносе наследуется от рабочего стенда, а что́ остаётся тестовым.

Параметры подключения к базе (имя, пользователь, пароль) хранятся в файле `data/config-internal.php` внутри тома EspoCRM. При переносе файлы рабочего стенда распаковываются в тестовый том вместе с этим файлом, поэтому тестовая копия будет пытаться подключиться к базе с параметрами рабочего стенда. Чтобы подключение сработало, тестовая база должна использовать те же имя, пользователя и пароль. Именно поэтому их и приравнивают к рабочим.

Адрес стенда (`siteUrl`) и адрес WebSocket — наоборот, остаются тестовыми автоматически. В compose-файле заданы переменные `ESPOCRM_CONFIG_SITE_URL` и `ESPOCRM_CONFIG_WEB_SOCKET_URL`, которые применяются при каждом запуске контейнера и берут значения из `.env.dev`. Поэтому даже после заливки рабочей копии тестовый стенд при старте выставит свой собственный адрес `http://...:8080`, а не рабочий `http://10.0.2.7`.

Если тестовый стенд уже был создан с другими паролями БД, проще пересоздать только тестовый стенд:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml down -v
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
```

Эта команда допустима только для тестового стенда. На рабочем стенде `down -v` не использовать.

## 4. Перенос рабочего стенда в тестовый

Эта процедура полностью пересоздаёт тестовый стенд свежей копией рабочего стенда. Тестовый стенд при этом стирается целиком: его контейнеры и тома удаляются и создаются заново, поэтому от прошлых экспериментов не остаётся ни старых данных, ни мусорных таблиц. Рабочий стенд не останавливается и не меняется.

Полное пересоздание возможно именно потому, что тестовый стенд одноразовый: его данные не нужно беречь. Для рабочего стенда так делать нельзя.

После переноса тестовый стенд можно использовать для экспериментов: проверять обновления, настройки, доработки и сценарии восстановления.

Перед началом убедитесь, что:

| Проверка | Зачем |
|---|---|
| Рабочий стенд запущен | с него снимается резервная копия |
| Данные тестового стенда не нужны | он будет полностью удалён и создан заново |
| В `.env.dev` БД совместима с prod | чтобы восстановленная копия подключилась к dev-БД |
| Есть образ `alpine` | он нужен для работы с файловым томом |

Шаги ниже используют переменные оболочки (`BACKUP`, `DEV_DB_CONTAINER` и другие). Выполняйте их в одной сессии терминала и без перерыва: при открытии нового терминала переменные теряются, и шаги нужно начинать сначала.

### Шаг 1. Сделать свежую резервную копию рабочего стенда

```bash
cd /opt/espocrm
mkdir -p backups
bash ./scripts/backup-docker-container.sh espocrm ./backups
```

Выбрать архив:

```bash
ls -lh backups/*.tar.gz
```

Задать переменную:

```bash
BACKUP=./backups/YYYY-MM-DD_HHMMSS.tar.gz
```

Проверить архив:

```bash
tar tzf "$BACKUP"
```

Если архив не читается, не использовать его для переноса.

### Шаг 2. Пересоздать тестовый стенд с нуля

Удаляем тестовые контейнеры вместе с их томами и поднимаем тестовый стенд заново. Главное здесь — получить чистую тестовую базу: старый том базы удаляется, и при старте контейнер создаёт пустую базу с пользователем и паролем из `.env.dev`, без следов прошлых экспериментов. Файловый том тоже пересоздаётся, но его содержимое позже всё равно будет заменено рабочей копией на Шаге 7, поэтому за файлы здесь беспокоиться не нужно.

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml down -v
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
```

Эта команда затрагивает только тестовый стенд. Рабочий стенд (`docker-compose.prod.yml`) не трогается.

Дождитесь готовности тестовой базы и приложения — первый старт на пустых томах идёт до двух минут:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml ps
```

У `espocrm-dev` и `espocrm-db-dev` в колонке `STATUS` должно стать `healthy`. Только после этого переходите к следующему шагу: тестовая база должна успеть создаться из `.env.dev`, иначе загрузка дампа не пройдёт.

### Шаг 3. Подготовить файлы из архива

```bash
rm -rf restore-tmp
mkdir restore-tmp
tar xzf "$BACKUP" -C restore-tmp
tar xzf restore-tmp/db.tar.gz -C restore-tmp
```

Проверить:

```bash
ls -lh restore-tmp
```

Должны быть:

```text
restore-tmp/db.sql
restore-tmp/files.tar.gz
```

### Шаг 4. Считать параметры тестовой БД

Параметры нужно брать из контейнера `espocrm-dev`, потому что восстанавливать будем именно в тестовую базу.

```bash
DEV_DB_CONTAINER=$(docker exec espocrm-dev printenv ESPOCRM_DATABASE_HOST)
DEV_DB_NAME=$(docker exec espocrm-dev printenv ESPOCRM_DATABASE_NAME)
DEV_DB_USER=$(docker exec espocrm-dev printenv ESPOCRM_DATABASE_USER)
DEV_DB_PASS=$(docker exec espocrm-dev printenv ESPOCRM_DATABASE_PASSWORD)
```

Проверить, что значения не пустые:

```bash
printf 'DEV_DB_CONTAINER=%s\nDEV_DB_NAME=%s\nDEV_DB_USER=%s\n' "$DEV_DB_CONTAINER" "$DEV_DB_NAME" "$DEV_DB_USER"
test -n "$DEV_DB_CONTAINER" && test -n "$DEV_DB_NAME" && test -n "$DEV_DB_USER" && test -n "$DEV_DB_PASS"
```

Если проверка не прошла, остановиться.

### Шаг 5. Остановить приложение тестового стенда

Останавливаем только тестовые контейнеры приложения. Тестовая база должна продолжать работать.

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml stop espocrm-daemon espocrm-websocket espocrm
```

Проверить, что база тестового стенда работает:

```bash
docker ps --format '{{.Names}}'
```

В списке должен быть:

```text
espocrm-db-dev
```

### Шаг 6. Залить рабочую базу в тестовую БД

```bash
cat restore-tmp/db.sql | docker exec -i "$DEV_DB_CONTAINER" \
  mariadb --user="$DEV_DB_USER" --password="$DEV_DB_PASS" "$DEV_DB_NAME"
```

Если команда завершилась с ошибкой, не запускать тестовый стенд. Сначала исправить ошибку загрузки БД.

### Шаг 7. Заменить файлы тестового стенда

После пересоздания на шаге 2 в файловом томе тестового стенда лежит чистая установка. Перед распаковкой рабочей копии её нужно убрать, чтобы файлы не смешались.

Очистить файловый том тестового стенда:

```bash
docker run --rm --volumes-from espocrm-dev alpine \
  sh -c 'rm -rf /var/www/html/* /var/www/html/.[!.]* /var/www/html/..?*'
```

Распаковать файлы из рабочей копии:

```bash
cat restore-tmp/files.tar.gz | docker run --rm -i --volumes-from espocrm-dev alpine \
  tar xzf - -C /var/www/html
```

### Шаг 8. Запустить тестовый стенд

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d
docker compose --env-file .env.dev -f docker-compose.dev.yml ps
```

Открыть тестовый стенд:

```text
http://10.0.2.7:8080
```

или, если работа идёт прямо на сервере:

```text
http://localhost:8080
```

### Шаг 9. Проверить результат

После переноса нужно проверить не только открытие страницы, но и данные.

| Что проверить | Ожидаемый результат |
|---|---|
| Вход администратора | вход работает |
| Основные разделы | данные похожи на рабочий стенд |
| Вложения | файлы открываются |
| Адрес в браузере | тестовый адрес, не рабочий |
| Рабочий стенд | продолжает открываться по `http://10.0.2.7` |

После переноса в тестовый стенд попадает база данных рабочего стенда. Поэтому вход в тестовую CRM выполняется с учётными данными из рабочего стенда.

`ESPOCRM_ADMIN_PASSWORD` из `.env.dev` используется только при первой установке пустого тестового стенда. После загрузки рабочей копии этот пароль уже не влияет на существующего администратора.

Если после переноса тестовый стенд всё-таки уводит на рабочий адрес `http://10.0.2.7`, значит контейнер ещё не переприменил `ESPOCRM_CONFIG_SITE_URL`. Пересоздайте контейнеры тестового стенда и проверьте, что в `.env.dev` указан тестовый `ESPOCRM_SITE_URL`:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml up -d --force-recreate
```

Другие возможные проблемы после переноса (вход, подключение к базе, WebSocket) разобраны в [`07-troubleshooting.md`](07-troubleshooting.md).

После проверки можно удалить временные файлы:

```bash
rm -rf restore-tmp
```

## 5. Остановить тестовый стенд

Обычная остановка тестового стенда не удаляет его данные:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml down
```

Полная очистка тестового стенда с удалением dev-томов:

```bash
docker compose --env-file .env.dev -f docker-compose.dev.yml down -v
```

Эту команду использовать только для dev. На рабочем стенде она запрещена.
